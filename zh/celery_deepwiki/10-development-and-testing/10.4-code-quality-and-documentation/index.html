    目的与范围¶

    本文档描述了 Celery 的代码质量强制执行和文档验证系统。它涵盖了静态分析工具（flake8, mypy, bandit）、文档完整性检查（apicheck, configcheck, linkcheck），以及用于交叉引用的自定义 celerydocs.py Sphinx 扩展。有关测试执行框架的信息，请参阅测试基础设施。有关 CI/CD 集成的详细信息，请参阅测试相关页面。

    Celery 通过多个验证层强制执行质量标准，在代码合并之前验证代码风格、类型正确性、安全性和文档完整性。

    Celery 的质量强制执行系统由多个独立的验证工具组成，每个工具验证代码质量的不同方面：

    代码风格检查 (flake8)¶

    flake8 环境负责在整个 Celery 代码库中验证代码风格、复杂度和 docstring 质量。

    Flake8 配置¶

    工具栈：

    • flake8>=3.8.3 - 核心 Linter requirements/pkgutils.txt3。
    • flake8-docstrings>=1.7.0 - Docstring 风格检查器 requirements/pkgutils.txt4。
    • pydocstyle==6.3.0 - Docstring 规范强制执行器 requirements/pkgutils.txt5。

    执行方法：

    # 通过 Makefile 执行
make flakecheck

# 通过 tox 执行（隐式环境）
tox -e lint

# 直接执行
flake8 celery t
    Makefile 集成 Makefile94-95：

    flakecheck:
    $(FLAKE8) "$(PROJ)" "$(TESTDIR)"
    这会同时检查 celery/ 源代码目录和 t/ 测试目录。

    来源： requirements/pkgutils.txt3-5 Makefile94-95 Makefile86

    Flake8 验证的内容¶

    代码风格检查：

    • 符合 PEP 8 规范（行号、缩进、空格）。
    • 命名约定（函数、类、变量）。
    • 导入顺序和组织。
    • 未使用的导入和变量。

    复杂度检查：

    • 函数的圈复杂度 (Cyclomatic complexity)。
    • 函数长度限制。
    • 嵌套深度限制。

    Docstring 检查：

    • 模块、类和函数 docstring 的存在性。
    • Docstring 格式 (PEP 257)。
    • 参数文档的完整性。

    来源： requirements/pkgutils.txt3-5

    静态类型检查 (mypy)¶

    mypy 环境 tox.ini101-102 执行静态类型分析，以便在运行时之前捕获与类型相关的错误。

    MyPy 配置¶

    Tox 环境定义：

    [testenv:mypy]
commands = python -m mypy --config-file pyproject.toml
    执行：

    tox -e mypy
    类型检查配置存储在 pyproject.toml 中，它定义了：

    • 被检查的模块和包。
    • 类型严格程度。
    • 忽略的错误或模块。
    • 插件配置。

    来源： tox.ini101-102

    类型检查范围¶

    MyPy 分析以下内容：

    • 函数和方法签名。
    • 变量类型注解。
    • 返回类型的正确性。
    • 赋值中的类型兼容性。
    • 泛型 (Generic type) 的使用。

    这可以捕获如下问题：

    • 向函数传递错误类型。
    • 返回不正确的类型。
    • 访问未定义的属性。
    • 不兼容的类型操作。

    来源： tox.ini101-102

    安全扫描 (bandit)¶

    bandit 环境 tox.ini118-120 扫描 Celery 代码库中的常见安全漏洞。

    Bandit 配置¶

    Tox 环境定义：

    [testenv:bandit]
commands =
    bandit -b bandit.json -r celery/
    执行：

    tox -e bandit
    扫描使用来自 bandit.json 的配置，并递归检查 celery/ 目录中的所有文件。

    来源： tox.ini44 tox.ini118-120

    执行的安全检查¶

    Bandit 识别潜在的安全问题：

    • 硬编码的密码或密钥。
    • SQL 注入漏洞。
    • Shell 注入风险。
    • 不安全的加密实践。
    • 不安全的反序列化。
    • 弱随机数生成。
    • 路径遍历漏洞。
    • XML 外部实体 (XXE) 问题。

    bandit.json 配置文件定义了：

    • 严重性阈值。
    • 排除的测试。
    • 忽略的文件模式。
    • 自定义安全配置文件。

    来源： tox.ini118-120

    Pre-commit 钩子集成¶

    lint 环境 tox.ini122-124 执行 pre-commit 钩子以进行全面的代码验证。

    Pre-commit 配置¶

    Tox 环境定义：

    [testenv:lint]
commands =
    pre-commit {posargs:run --all-files --show-diff-on-failure}
    执行：

    # 在所有文件上运行所有钩子
tox -e lint

# 运行特定的钩子
tox -e lint -- run --hook-id flake8

# 仅在暂存文件上运行（典型的 pre-commit 用法）
pre-commit run
    来源： tox.ini43 tox.ini122-124

    Pre-commit 钩子¶

    .pre-commit-config.yaml 文件定义了自动运行的钩子：

    • 代码格式化器（行尾空格、EOF 修复器）。
    • Linter（flake8，以及其他可能的工具）。
    • 安全扫描器。
    • 导入排序器。
    • YAML/JSON 验证器。

    这些钩子运行的时机：

    1. 自动：在 git commit 时运行（如果已安装 pre-commit）。
    2. CI 中：通过 lint tox 环境运行。
    3. 手动：通过 tox -e lint 运行。

    来源： tox.ini122-124

    文档质量验证¶

    Celery 使用自定义的 Sphinx 构建器来确保文档的完整性和准确性。

    API 覆盖率验证 (apicheck)¶

    apicheck 构建器验证所有公共 API 元素是否都已记录在参考文档中。

    API 覆盖率检查流程：

    flowchart TD
    APICheckIgnored["apicheck_ignore_modulesIgnored Modules List"]
    DocsConf["docs/conf.pySphinx Configuration"]
    CeleryPackage["celery/ PackageAll Python Modules"]
    IntrospectModules["sphinx.ext.autodocModule Introspection"]
    ExtractAPIs["Extract Public APIsClasses, Functions, Methods"]
    APIReference["docs/reference/API Documentation"]
    DocumentedAPIs["Documented API ElementsFrom RST Files"]
    CompareAPIs["Compare SetsCode vs Docs"]
    MissingAPIs["Report Missing APIsExit with Error"]
    Success["All APIs DocumentedBuild Success"]

    APICheckIgnored --> IntrospectModules
    DocsConf --> IntrospectModules
    CeleryPackage --> IntrospectModules
    IntrospectModules --> ExtractAPIs
    APIReference --> DocumentedAPIs
    ExtractAPIs --> CompareAPIs
    DocumentedAPIs --> CompareAPIs
    CompareAPIs --> MissingAPIs
    CompareAPIs --> Success
    执行方法：

    # 通过 tox
tox -e apicheck

# 通过根目录 Makefile
make apicheck

# 通过 docs 目录下的 Makefile
cd docs && make apicheck

# 直接调用 Sphinx
sphinx-build -b apicheck -d docs/_build/doctrees docs docs/_build/apicheck
    Tox 环境 tox.ini104-108：

    [testenv:apicheck]

    版本特定的依赖项¶

    依赖系统因 Python 版本而异 tox.ini35-37：

    CPython 3.9-3.14:

    • 完整的 CI 依赖：requirements/test-ci-default.txt。
    • 文档工具：requirements/docs.txt。

    PyPy3:

    • 仅 CI 依赖：requirements/test-ci-default.txt。
    • 无文档依赖（文档构建需要 CPython）。

    这种优化减少了 PyPy 环境的设置时间，因为文档仅在 CPython 上构建。
