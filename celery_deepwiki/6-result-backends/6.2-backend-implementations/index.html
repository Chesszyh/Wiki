    This document covers the specific implementations of result backends in Celery. These implementations provide the concrete functionality for storing, retrieving, and managing task results using different storage systems. For the overall backend architecture and interfaces, see Result Backend Architecture. For working with results from a user perspective, see Working with Results.

    Backend Implementation Overview¶

    Celery provides multiple backend implementations that extend the base backend architecture to support different storage systems. Each implementation provides specific capabilities and trade-offs for result storage and retrieval.

    MongoDB Backend Operations¶

    The MongoBackend class celery/backends/mongodb.py35-334 implements document-based result storage:

    Core Operations:

    • _store_result() - Store task result using replace_one() with upsert celery/backends/mongodb.py181-194
    • _get_task_meta_for() - Retrieve task metadata using find_one() celery/backends/mongodb.py197-224
    • _save_group() - Store group result celery/backends/mongodb.py226-234
    • _restore_group() - Retrieve group result celery/backends/mongodb.py236-247
    • _delete_group() - Delete group by ID celery/backends/mongodb.py249-251
    • _forget() - Remove result using delete_one() celery/backends/mongodb.py253-263
    • cleanup() - Delete expired documents celery/backends/mongodb.py265-275

    Document Structure: Each task result is stored as a MongoDB document with _id as the task ID celery/backends/mongodb.py188 and includes standard metadata fields like status, result, date_done, and traceback. Extended metadata includes name, args, kwargs, queue, worker, and retries celery/backends/mongodb.py201-215

    Binary Codec Handling: The backend handles serialization through BINARY_CODECS = frozenset(['pickle', 'msgpack']) celery/backends/mongodb.py32 and wraps binary data with Binary() for MongoDB storage celery/backends/mongodb.py172-173

    Connection Management:

    • _get_connection() - Creates MongoClient with connection options celery/backends/mongodb.py136-163
    • _ensure_mongodb_uri_compliance() - Validates and normalizes MongoDB URIs celery/backends/mongodb.py118-127
    • _prepare_client_options() - Configures client options based on PyMongo version celery/backends/mongodb.py129-134

    Cassandra Backend Features¶

    The CassandraBackend class celery/backends/cassandra.py71-257 provides distributed result storage:

    Feature Support:

    • supports_autoexpire = True - TTL-based automatic expiration celery/backends/cassandra.py86
    • Dual connection modes: standard clusters or AstraDB cloud bundles celery/backends/cassandra.py104-110
    • Consistency level configuration for read/write operations celery/backends/cassandra.py117-125

    Query Templates: The backend defines CQL query templates as constants celery/backends/cassandra.py37-64:

    • Q_INSERT_RESULT - Task result insertion with optional TTL
    • Q_SELECT_RESULT - Task result selection by task_id
    • Q_CREATE_RESULT_TABLE - Table creation with clustering order
    • Q_EXPIRES - TTL clause template using USING TTL {0}

    Table Schema: The backend creates tables using Q_CREATE_RESULT_TABLE celery/backends/cassandra.py50-60:

    CREATE TABLE {table} (
    task_id text,
    status text,
    result blob,
    date_done timestamp,
    traceback blob,
    children blob,
    PRIMARY KEY ((task_id), date_done)
) WITH CLUSTERING ORDER BY (date_done DESC);
    Connection Management:

    • _get_connection() handles thread-safe connection setup with threading.RLock() celery/backends/cassandra.py142-212
    • Standard clusters created with servers list celery/backends/cassandra.py155-159
    • AstraDB cloud uses secure_connect_bundle from bundle_path celery/backends/cassandra.py161-167
    • Authentication via cassandra.auth providers celery/backends/cassandra.py128-134

    Operations:

    • _store_result() executes prepared write statement with buf_t() encoding celery/backends/cassandra.py214-226
    • _get_task_meta_for() uses prepared read statement returning meta dictionary celery/backends/cassandra.py231-248
    • Auto-creates result table on first write operation celery/backends/cassandra.py183-200

    Sources: celery/backends/cassandra.py37-64 celery/backends/cassandra.py86 celery/backends/cassandra.py142-248

    Elasticsearch Concurrency Control¶

    The ElasticsearchBackend class celery/backends/elasticsearch.py30-284 implements sophisticated concurrency control:

    Optimistic Locking Implementation: The _update() method celery/backends/elasticsearch.py167-231 uses Elasticsearch's optimistic concurrency control:

    • Retrieves current _seq_no and _primary_term from existing document celery/backends/elasticsearch.py201-202
    • Updates with if_primary_term and if_seq_no parameters celery/backends/elasticsearch.py211 celery/backends/elasticsearch.py219
    • Raises ConflictError if sequence number/primary term mismatch celery/backends/elasticsearch.py226-230

    State Validation Logic: The _update() method implements state protection celery/backends/elasticsearch.py187-197:

    • Decodes existing result with decode_result() to check current state celery/backends/elasticsearch.py188
    • Returns {'result': 'noop'} if existing state is SUCCESS celery/backends/elasticsearch.py192-194
    • Prevents UNREADY_STATES from overwriting READY_STATES celery/backends/elasticsearch.py195-197

    Document Operations:

    • _set_with_state() attempts _index() with op_type: 'create', falls back to _update() on conflict celery/backends/elasticsearch.py128-142
    • _index() handles both regular and doc_type modes celery/backends/elasticsearch.py147-165
    • Document structure includes @timestamp field with UTC timezone celery/backends/elasticsearch.py131-133

    Serialization Modes:

    • es_save_meta_as_text configuration controls serialization celery/backends/elasticsearch.py87
    • encode()/decode() methods handle both text and object modes celery/backends/elasticsearch.py233-255
    • Object mode preserves result and traceback as nested objects celery/backends/elasticsearch.py239-243

    Exception Handling:

    • exception_safe_to_retry() method identifies retryable exceptions celery/backends/elasticsearch.py90-102
    • Handles ApiError status codes (401, 409, 500, 502, 504) and TransportError celery/backends/elasticsearch.py91-101

    Sources: celery/backends/elasticsearch.py87 celery/backends/elasticsearch.py90-102 celery/backends/elasticsearch.py128-142 celery/backends/elasticsearch.py167-231 celery/backends/elasticsearch.py233-255

    Backend Configuration and Selection¶

    Different backends are selected via the result_backend configuration setting using URL schemes or backend names:

    Backend URL Schemes¶

    Backend URL Scheme Example
    Redis redis:// redis://localhost:6379/0
    MongoDB mongodb:// mongodb://user:pass@host/database
    Cassandra cassandra:// cassandra:// (uses config settings)
    Elasticsearch elasticsearch:// elasticsearch://localhost:9200/index

    Memory and Resource Usage¶

    Cache Backend:

    • Uses _DUMMY_CLIENT_CACHE = LRUCache(limit=5000) celery/backends/cache.py25 for in-memory storage
    • Shared between threads for memory efficiency
    • Supports connection pooling through cache client libraries

    RPC Backend:

    • Maintains _out_of_band dictionary for buffering unexpected results celery/backends/rpc.py124
    • Uses BacklogLimitExceeded protection with default limit of 1000 messages celery/backends/rpc.py232
    • One queue per client process via app.thread_oid celery/backends/rpc.py340-342
